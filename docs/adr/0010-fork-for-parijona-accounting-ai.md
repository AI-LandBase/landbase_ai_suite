# ADR 0010: parijona 向け AccountingAI を fork で独立進化させる

## ステータス

採用（Accepted）

## 日付

2026-07-02

## コンテキスト（背景・課題）

株式会社パリジョナ（parijona）から AccountingAI の要件定義書を受領した（発行 2026-07-01、発注元: 山内友理氏、開発: 株式会社アイランドベース）。要件定義書は「経理代行（記帳代行）業務のためのシステム」として TO-BE を定義し直したもので、現 `landbase_ai_suite` の実装との差分を多数含む。主な差分は次のとおり:

- **マネーフォワード（MF）形式 CSV 出力**の新規実装（最重要の不足。付録 A・B に仕様）
- **ナレッジ DB**（科目判定ルール）の 3 つの育成導線（CSV インポート / 事前登録 UI / 運用学習）の実装
- **証憑ビューア**（左右並列・複数枚・拡大縮小・回転）への拡張
- **Google ドライブ保管**（顧客別フォルダ自動作成、操作はシステム内で完結）
- **LINE 紐付けの是正**（単一既定顧客への自動割当 → 事前登録 / 後付け紐付け + 修正 UI、多対一）
- **電子帳簿保存法（電帳法）対応のリネーム**（検索要件を満たす方向。金額を含める `YYYYMMDD_支払先_金額_内容`）
- **進捗ステータス 3 軸方式**（進捗 3 段階 × review_required × exported_at）の新設
- 顧客マスタへの「規定の会計ソフト」「共有ドライブ URL」カラム追加
- 清掃マニュアル・観光業向け機能の削ぎ落とし（要件定義書 §2.3 / §5）

これらは AccountingAI（経理ドメイン）に特化した大規模改修であり、parijona 専用の色が濃い。一方で `landbase_ai_suite` は自社・既存クライアント向けの経理機能を現に運用しており、この改修をそのまま本リポに取り込むと責務とリリースサイクルが混ざる。

### ADR 0008 / 0009 との関係

- **ADR 0008（リポジトリ分割）は撤回（Withdrawn, 2026-06-21）**。物理分割ではなくモノレポ + モジュール境界で整理する方針へ転換済み。今回の決定は 0008 の復活（移設・分割）ではなく、**fork による別リポ派生**という別判断である。
- **ADR 0009（AccountingAI 機能拡張スコープ, 2026-06）**は、要件定義書によって一部判断が見直された（後述「決定 4」）。0009 の残スコープは要件定義書ベースで fork 先に実装する。

## 検討した選択肢

### 選択肢 1: 現リポにそのまま実装する（モノレポ継続）

**メリット**:
- 追加のリポ整備コストゼロ

**デメリット**:
- ❌ parijona 専用の大規模改修と自社運用の経理機能が同一リポで混在し、責務・リリースサイクル・アクセス管理が混ざる
- ❌ 要件定義書は「顧客接点は LINE のみ」「実データを扱うのは parijona の会計士・税理士」という parijona 固有の事業前提に立っており、自社運用の前提とズレる

**不採用の理由**: 上記の混在が解消しない。

### 選択肢 2: ADR 0008 のリポジトリ分割（移設）を復活させる

**デメリット**:
- ❌ 0008 は設計思想の変更で撤回済み。経理機能を本リポから抜くことは「自社・既存クライアント用に経理機能が必要」という現状と矛盾する
- ❌ 移設は本リポから経理を失う。両者に経理が必要な今回の状況に合わない

**不採用の理由**: 移設は要件に合わない。本リポは経理機能ごと存続させたい。

### 選択肢 3: fork して独立進化（採用）

- `landbase_ai_suite` を fork して parijona 向け新リポを作る
- 要件定義書の実装はすべて fork 先で行う
- 本リポは経理機能ごと現状維持で存続

**メリット**:
- ✅ 本リポは経理機能を失わず存続（自社・既存クライアント用）
- ✅ parijona 専用改修を fork 先に隔離でき、責務・リリース・アクセス管理を分離
- ✅ 移設に伴う大掛かりなデータ・インフラ移管が不要
- ✅ 欲しい改善だけを任意で cherry-pick でき、同期義務を負わない

**デメリット**:
- ⚠️ 共通基盤の改善が両リポに自動で伝播しない（cherry-pick は任意・手動）
- ⚠️ 新リポの初期整備（README / CLAUDE.md / CONTRIBUTING.md / ARCHITECTURE.md の書き直し、CI、デプロイ）が必要

## 決定

### 1. 構図は「fork して独立進化」（移設ではない）

- `landbase_ai_suite` を fork して parijona 向け新リポを作る
- 要件定義書の実装（MF 形式 CSV、ナレッジ DB、証憑ビューア、Google ドライブ保管、LINE 紐付け是正、電帳法リネーム、進捗ステータス 3 軸 等）は**すべて fork 先で行う**
- 同期義務なし。欲しい改善だけ cherry-pick（任意）

### 2. 旧リポ（landbase_ai_suite）は経理機能ごと現状維持で存続

- 自社・既存クライアント用に経理機能が必要なため削除しない
- ADR 0008 Phase 3 のような軽量化（rails/ や経理 skills の削除）はしない

### 3. 新リポ名は parijona 色を出す

- 仮称 `parijona_accounting_ai`（org は `AI-LandBase` 想定。要件定義書ヘッダーの `AI-LandBase/landbase_ai_suite` に準拠）
- 正式名・org 配置は未確定 → 「未決事項」参照

### 4. ADR の扱い

- ADR 0008（リポ分割）は Withdrawn のまま触らない。今回は 0008 の復活ではなく別判断（fork）
- **要件定義書は ADR 0009 の一部判断を見直している**。見直し点を本 ADR に明記する:
  - **リネームの主目的の変更**: 0009 では「画面表示で分かりやすくするため」（§G）だったが、要件定義書 §4.8 では「電帳法対応（検索要件を満たす）」が主目的。ファイル名は金額を含める方向（`YYYYMMDD_支払先_内容` → `YYYYMMDD_支払先_金額_内容`）
  - **クレカ明細突合の格下げ**: 0009 の⑦（クレカ末尾警告 = 領収書 × カード明細の二重計上検知に相当）は、要件定義書 §4.7(b) で**今回スコープ外（将来拡張）**へ格下げ。ただし土台（payment_cards・card_last_four・AMEX 明細処理）は温存する
  - **進捗ステータス 3 軸方式の新設**: 進捗 3 段階（未処理 → 処理中 → 確定） × review_required（既存） × exported_at（既存）を独立軸として管理（要件定義書 §4.8）
- ADR 0009 の残スコープは要件定義書ベースで fork 先に実装する

## 結果

### 期待効果

1. **責務分離**: parijona 専用の大規模改修が fork 先に隔離され、本リポの経理機能と混ざらない
2. **本リポの存続**: 自社・既存クライアント向け経理機能を失わない
3. **リリース独立**: 両リポが独立したリリースサイクル・アクセス管理を持てる
4. **改善の選択的取り込み**: 共通基盤の改善は cherry-pick で任意に取り込める

### トレードオフ

- ⚠️ 共通基盤の改善が両リポに自動伝播しない（同期は手動・任意）
- ⚠️ 新リポの初期整備コスト（ドキュメント書き直し、CI、デプロイ、清掃・観光系の削ぎ落とし）

### 関連 ADR への影響

| ADR 0008（リポジトリ分割） | Withdrawn のまま。今回は復活ではなく別判断（fork）である旨を本 ADR に記録 |

## 未決事項（Future Work）

- **新リポの正式名**（仮称 `parijona_accounting_ai`）
- **org の配置**: `AI-LandBase` に置くか、parijona 用 org を新設するか（アクセス管理の観点は ADR 0008 の未決事項欄も参照）
- **要件定義書 §7 の協議事項**（fork 先での設計課題。旧リポ側の整理には影響しない）:
  - Google ドライブ連携方式（サービスアカウント / 共有ドライブ構成、ActiveStorage との二重保管か置換か）
  - 電帳法対応範囲の税理士確認（検索要件の実装方法、真実性確保の手段、保存期間）
  - ナレッジ運用学習の判定条件（どの確定仕訳を、どのキーで学習するか）
  - 中間の汎用会計データの列設計（弥生・MF 双方へ過不足なく変換できるか）
