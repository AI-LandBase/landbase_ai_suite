# ADR 0009: AccountingAI 機能拡張のスコープと優先順位（2026-06）

## ステータス

採用（Accepted）

## 日付

2026-06-08

## コンテキスト（背景・課題）

特定の経理業務運用クライアント 1 社からの要望を起点として、AccountingAI に対する機能拡張要望が提示された。要件項目ごとに既存実装との差分を洗い出して合意形成を行い、本 ADR はその確定スコープ・設計判断の根拠・issue 分割方針を記録する。

**実装方針**: マルチテナント基盤上の汎用機能として実装し、要望元クライアント以外のテナントにも同じ機能を提供できる形を保つ。クライアント固有のカスタマイズではなく、AccountingAI 全体の機能拡張として位置づける。

### 要望の概要

クライアントから提示された要望は次の 5 領域にまたがる:

1. **業種登録**: 任意登録 / 多業種選択
2. **仕訳台帳の詳細ページ**: 削除・編集機能の追加、編集履歴の記録、証憑表示
3. **領収書アップロード機能の Web UI 化**: 複数ファイル同時アップ、現金/クレカ判定、クレカ末尾一致による重複警告、項目別仕訳
4. **ファイル命名規約**: `YYYYMMDD_支払先_内容` 形式
5. **クレカマスタ管理**: カード末尾の事前登録

### 既存実装との差分（要約）

- 業種は `clients.industry`（単一 string、`restaurant / hotel / tour`）。多業種未対応
- 仕訳の削除アクション/ルートは未実装、編集ボタンは `review_required` ステータスのみに表示、編集履歴は記録機構なし
- 詳細ページから証憑（`statement_batch.pdf`）への参照リンクなし
- 領収書アップロードは LINE Webhook 経由のみ。Web UI は未実装
- 領収書処理プロンプトは「1 領収書 = 1 取引」前提で、項目別仕訳は未対応（ただし `JournalEntryCreator` の保存ループ自体は複数取引対応済み）
- ファイル名は LINE 経由が `receipt_{message_id}.jpg`、その他は元ファイル名のまま。統一規約なし
- クレカ末尾を保持するマスタ・末尾一致による重複検知ロジックなし

## 決定

### 確定スコープ

| 領域 | 確定仕様 |
|---|---|
| A. 業種 | `clients.industries`（複数選択可）に変更。**まずは PostgreSQL の string 配列カラムで実装**し、将来選択肢が動的に増える運用が発生した時点で `industries` マスタテーブル + 中間テーブルへ移行できるよう、`Client#industry_codes` 等のアクセサで実装詳細を隠蔽する |
| B. 削除 | 仕訳 1 件削除 + バッチ単位削除の両方。二重アラート |
| C. 編集 | 全ステータスで編集可能化。編集日時・編集者・変更箇所（差分）・変更理由（任意）をリビジョンとして保存。詳細ページ内の履歴タブで表示 |
| D. 証憑表示 | 詳細ページに `statement_batch.pdf` を表示。リンク先は ActiveStorage の blob URL |
| E. クレカ末尾警告 | クレカ末尾の事前登録 UI + 領収書取込時に末尾一致で**赤ラベル表示**（②方式）。手動削除判断 |
| F. 項目別仕訳 | 領収書の購入品目ごとに仕訳を分割。項目別仕訳が発生した領収書は自動で `review_required` を付与 |
| G. リネーム規約 | `YYYYMMDD_支払先_内容` 形式で **ActiveStorage 保存時にリネーム**。1 ファイル = 1 リネーム、代表内容を採用 |

### issue 分割と依存関係

| # | issue 内容 | 規模 | 依存 |
|---|---|---|---|
| ① | 業種マルチ選択化 | 小 | なし |
| ② | 仕訳台帳: 削除機能（1 件 + バッチ単位、二重アラート） | 小 | なし |
| ③ | 仕訳台帳: 編集履歴（全ステータス編集 + リビジョン保存 + 履歴タブ） | 大 | なし |
| ④ | 詳細ページに証憑表示 | 小 | なし |
| ⑤ | ファイル保存時のリネーム規約適用 | 中 | なし |
| ⑥ | 領収書 Web UI（複数ファイル同時アップ + 項目別仕訳 + 要確認自動付与） | 大 | ⑤ |
| ⑦ | クレカマスタ管理 UI + 領収書取込時の末尾警告 | 中 | ⑥ |

依存チェーン: `①〜④` は独立並行可、`⑤ → ⑥ → ⑦` は順次。

### 推奨着手順

1. `②④`（小・即効性）
2. `①`（独立）
3. `⑤⑥⑦`（依存チェーン）
4. `③`（リビジョン保存方式の設計判断を要するため最後）

## 設計判断の根拠

### E. クレカ重複警告を「赤ラベル方式（②）」とした理由

候補は 2 つあった:

- **① 取込時に弾く**: 領収書アップロード時点で重複を検知し、保存を拒否する
- **② 赤ラベル方式**: 仕訳として保存した上で、台帳の一覧/詳細に赤ラベルを表示し、担当者が手動で削除判断する

**② を採用**した理由:

- ①は誤検知時のリカバリーが重い。「日付・金額・末尾一致」だけでは別取引の偶然一致を排除できず、本当に重複か領収書を確認する前にデータが消えるとやり直し工数が大きい
- ②なら、台帳上で AMEX 明細仕訳と並べて目視確認してから削除判断できる
- クライアントの当初メモにも「警告ラベル（赤）（削除推奨）」とあり、自動削除ではなく担当者判断を前提とした表現になっている

**実装側で詰める前提**: 赤ラベルの具体的な表示箇所（仕訳一覧の特定カラム、詳細ページの上部バナー、両方など）と UI コンポーネントの詳細は、⑦ クレカマスタ管理 UI + 末尾警告の実装着手時に別途決定する。

### F. 項目別仕訳に「`review_required` 自動付与」を採用した理由

- 単一合計の仕訳と異なり、品目別に勘定科目を推定する処理は AI の判定精度が落ちる
   - 例: 「食材 + 日用品」のレシートで全行を `仕入高` にまとめてしまうケースが想定される
- 精度に自信が持てない仕訳は既存仕様（`status: review_required`）と同じく担当者レビューに回す方が安全
- `review_required` は既に台帳の編集導線と紐付いており、追加実装コストが低い

### G. リネームを「ActiveStorage 保存時」に行う理由

クライアントから「リネーム規約は画面表示で分かりやすくするため」との回答を受けた。よって:

- ダウンロード時の動的リネームではなく、保存時点でファイル名を確定させる
- `active_storage_blobs.filename` を規約に従って書き換えれば、画面上のファイル一覧・ダウンロード時の双方で一貫した名前になる

### ④ クレカ末尾を「事前登録方式」とした理由

AMEX 明細 PDF からカード末尾を抽出することは技術的には可能だが:

- マスタを事前登録する方が、利用カードが追加された際の検知漏れ・誤抽出の影響を局所化できる
- クライアント側から「必要であれば事前に登録する」と回答があり、運用負担を許容する意思が確認できた
- 警告判定ロジックがマスタ参照のみで完結し、明細処理パイプラインへの依存が減る

### 弥生 CSV 税区分マッピングの暫定方針（PR #309 / issue#308 参照）

CSV 出力時に `tax_category` 文字列を弥生の 3 区分（`課対仕入込10%` / `課対仕入込軽減8%` / `対象外`）へマッピングする処理を `YayoiExportService` に実装した。本 ADR で扱う機能拡張群の前提となるため、暫定方針として記録する。

**マッピングルール（PR #309 時点）**:

| DB の `tax_category` | CSV 出力 |
|---|---|
| `10%` を含む（例: `課税仕入10%（インボイス）`） | `課対仕入込10%` |
| `8%` または `軽減` を含む | `課対仕入込軽減8%` |
| その他（`対象外` / `非課税仕入` / `課対仕入（リバースチャージ）` 等） | `対象外` |
| 空文字 | 空文字（弥生側の判定に委ねる） |

**暫定であることの明示**:

- 適格/非適格サフィックスは付与せず、弥生側の自動判定に委ねている
- リバースチャージ・非課税仕入を `対象外` に寄せているのは、**3 区分に当てはまらないものを一旦集約した暫定分類**である。弥生公式マスタには `課対仕入(リバースチャージ)` 等の正式区分が別途存在する可能性が高く、本来はそれらへ振り直すべき
- 上記の理由から、本マッピングは**運用フィードバックを受けて見直す前提**で運用する

**後続検討の方針**:

- 税率追加・新区分対応が発生した場合、`include?` ベースの分岐を増やすのではなく、**弥生公式マスタとの対応表（マッピングテーブル方式）**への置き換えを検討する
- 「税率 10% と 軽減 を両方含む文字列」のような曖昧なケースは、現状の優先順位（`10%` 判定が `軽減` より先）で判定される。仕様として確定させる場合は対応表化と同時に行う

**将来の対応表方式のスケッチ**:

将来的なマッピングテーブル化のイメージとしては、`tax_categories` マスタテーブルに次のような列を持たせる構造を想定する:

| カラム | 役割 |
|---|---|
| `tax_code` | 税区分コード（例: `T10` / `T08R` / `NT` / `OS`） |
| `tax_name` | 内部名称（例: `課税10%` / `軽減8%` / `非課税` / `不課税`） |
| `tax_rate` | 税率（例: `0.10` / `0.08` / `0.00`） |
| `tax_type` | 種別（`taxable` / `exempt` / `out_of_scope`） |
| `yayoi_label` | 弥生 CSV 出力時の文字列（例: `課対仕入込10%`） |

これにより `journal_entry_lines.tax_category` を生文字列ではなく `tax_code` 参照に正規化し、CSV 出力時はマスタの `yayoi_label` を引くだけで弥生形式へ変換できる。本 ADR ではあくまで方向性のスケッチであり、移行スコープ・移行手順は別 ADR で扱う。

### CSV出力済みフラグの暫定方針（PR #320 / issue#319 参照）

仕訳台帳の「CSV未出力だけ絞り込む」運用要望に対し、`journal_entries.exported_at`（datetime, nullable）を追加してフラグ方式で実装した。NULL = 未出力、値あり = 直近のCSV出力日時。Web/API の index/export 両方で `csv_export_status=unexported|exported` の絞り込みに対応する。

**運用上の仕様**:

- 再エクスポート時は **既存の `exported_at` を最新日時で上書き** する。過去の出力履歴は保持しない
- これは「直近いつ出力したか」だけ追える運用を前提とした割り切り。複数回出力したケースで「初回出力日時」は失われる
- 不正な `csv_export_status` 値（typo 等）は silent acceptance で全件を返す。`source_type` フィルタと同じ挙動

**将来の検討事項**:

- 「いつ・誰が・何形式で出したか」の監査要件が出た時点で、別 ADR / issue で `journal_entry_exports` のような履歴テーブル化を検討する
- それまでは本フラグ方式で運用フィードバックを集める
- `csv_export_status=exported` が高頻度なクエリになる場合は、現状の `WHERE exported_at IS NULL` 部分インデックスに加え、`IS NOT NULL` 側のインデックスも検討する

## 採用しなかった案

### クレカ警告: 自動削除

メモの初期表現には「削除推奨」とあったが、自動削除は採用しない。理由は E-2 の選択理由と同じく、誤検知時のリカバリー困難性。

### 業種: 単一選択維持

複数業種を運営するクライアントの存在が確認できたため、単一選択維持は不採用。`INDUSTRY_FEATURES`（業種→機能マップ）は OR 集約方式に変更する想定。

### 編集履歴: 履歴は記録のみで画面表示しない

クライアントから「詳細ページ内タブで表示」との明確な要望があったため、画面表示は必須。

### リネーム: ダウンロード時の動的リネーム

ActiveStorage 保存時にリネームする方式に統一。動的リネームは責務が分散し、画面表示との整合性が取りにくいため不採用。

## 影響

### 実装に与える影響

- `clients.industry`（string）→ `clients.industries`（string 配列）へのマイグレーションが必要
   - **既存値の扱い**: 単一 string 値は **1 要素配列に変換** してマイグレートする（例: `"restaurant"` → `["restaurant"]`）。`industry` カラム削除は移行 PR が安定するまで保留
- `journal_entries` の編集履歴を保持する新規テーブル（`journal_entry_revisions` 想定）が増える
- `ReceiptProcessorService` のシステムプロンプトを「項目別仕訳」対応に書き換える。LINE 経由ジョブ（`ReceiptLineProcessJob`）と Web UI 経由ジョブで同じプロンプトを共有する
- クレカマスタ用の新規モデル（`payment_cards` 想定）と関連 UI が増える
- ActiveStorage 保存時のフック（`Blob#filename` の書き換え）を統一実装する必要がある
   - **既存 blob の扱い**: 既に保存されている blob のリネームバックフィルは ⑤ のスコープ外。**新規保存分のみ規約適用** とし、既存ファイルのバックフィルは別タスクで判断する（バッチ処理 or バックフィルしない）

### 既存仕様との関係

- ADR 0005（マルチテナント戦略）の `client_code` 論理分離は維持
- ADR 0008（リポジトリ分割）における AccountingAI の責務範囲内で完結
- 仕訳の貸借一致バリデーション（`JournalEntry#amounts_must_balance`）は項目別仕訳でも維持。借方合計 = 貸方合計のルールは品目分割後も適用される

### 後続検討事項

- 編集履歴の保存方式（PaperTrail / audited / 自作テーブル）は ③ 着手時に別途検討
- 業種マスタの拡張運用（マスタテーブル化するか、定数で管理するか）は ① 着手時に決定
- リネーム規約の「冗長すぎを避ける」具体的な文字数上限は ⑤ 着手時に決定
- PR #309 マージ後、**実運用環境またはステージングで弥生インポートの動作確認**を 1 回取る
- 税区分マッピングを**弥生公式マスタとの対応表方式**へ置き換える時期・スコープは運用フィードバック後に判断
